## 21. `study-run-baseline` requires `@composite_generator`, not `*.composite.yaml`

I had three perfectly valid composite specs at
`pbg_membrane_actin_composite/composites/rung{1,2,3}_*.composite.yaml`,
each generated by `scripts/regen-composites.py` and discovered by
`pbg_superpowers.composite_discovery` (the Composites tab listed them).
But `POST /api/study-run-baseline` returned:

> "composite 'pbg_membrane_actin_composite.composites.rung1_fixed_boundary' not in generator registry"

The dashboard's run handler uses `pbg_superpowers.composite_generator._REGISTRY`
(populated by `@composite_generator` decorators), NOT `composite_discovery`
(which reads YAML files). Two parallel registries with similar names. The
Composites tab lists both; the Run handler only accepts decorator-based.

**Recommendation:** Either unify the registries or make the Run handler
fall back to the spec-based registry, instantiating composites from YAML
when no generator is registered. Currently a user could ship a workspace
that *looks* runnable (Composites tab populated) but isn't.

## 22. Top-level package must import subpackages for `@composite_generator` to register

`@composite_generator` decorators only fire when their containing module
is imported. The dashboard does
`importlib.import_module(workspace_package)` — which loads
`pbg_membrane_actin_composite/__init__.py` only. My generators in
`composites/__init__.py` were invisible until I added an explicit
`from . import composites` at the top of `pbg_membrane_actin_composite/__init__.py`.

**Recommendation:** Document this in the `@composite_generator` docstring.
Better, have `discover_generators()` walk the workspace package's
subpackages (using `pkgutil.walk_packages`) so subpackages don't need to
be eagerly imported from `__init__.py`.

## 23. ReaDDy/Mem3DG C++ libc stdout writes pollute the subprocess result pipe

The single thorniest bug of the session. The dashboard's `_run_composite_subprocess`
runs:

```python
result = subprocess.run([py, "-c", script], capture_output=True, text=True, ...)
payload = json.loads(out.split("@@@RESULTS@@@", 1)[1].strip(), object_hook=...)
```

When the script runs a composite containing a ReaDDy process, ReaDDy's C++
spdlog writes thousands of "Simulation completed" lines to OS fd 1 via libc.
The Python script writes `@@@RESULTS@@@\n{JSON}` via `print()`. Order of
flushes at process exit:

1. Python flushes its stdout buffer → fd 1 (the pipe) gets the marker + JSON.
2. C++ libc flushes its own buffered stdout → fd 1 gets thousands of trailing
   "Simulation completed" log lines.

Result: `out.split("@@@RESULTS@@@", 1)[1]` is the JSON followed by ReaDDy
logs. `json.loads` rejects trailing data with "Extra data" — error message
surfaces as "could not parse run output". The subprocess actually succeeded;
the dashboard just can't read the answer.

**Workaround in this workspace** (`pbg_membrane_actin_composite/core.py`):

```python
def build_core():
    _redirect_cxx_logs_to_stderr_if_piped()  # one-shot fd 1 → fd 2 redirect
    ...
```

Mechanism: dup the original fd 1 to a new fd (preserving Python's
stdout pipe), then `os.dup2(2, 1)` so OS-level fd 1 is now stderr.
Python's `print()` still writes to the saved fd; C++ writes go to stderr.

Two subtleties that ate 30+ minutes:

- **Use a MODULE-level Python flag**, not `os.environ`. An env-var flag
  leaks into every child subprocess via `os.environ` inheritance — so only
  the first subprocess redirects, every subsequent run fails the same way.
  A module-level boolean is process-local: each fresh subprocess starts
  with the flag clear and re-runs the redirect.
- **Don't `os.dup2(2, 1)` and forget about sys.stdout.** After the dup2,
  Python's sys.stdout still points at fd 1 — which is now stderr. You
  have to rebind sys.stdout to a fdopen() on the SAVED fd before the
  redirect, so Python prints continue going to the pipe.

**Recommendation for the upstream framework:** This bug will hit *every*
workspace that wraps a C++ simulator. Push the fix into pbg-superpowers
(or vivarium-dashboard's subprocess runner): wrap the generated script in
a stdout-fd preservation pattern. Or change the result-parsing to use a
DELIMITED-JSON convention (e.g., `@@@RESULTS@@@<base64-json>@@@END@@@`)
that's robust to trailing garbage.

## 24. `inject_sqlite_emitter` matches by `addr.endswith("Emitter")` — case-sensitive

The dashboard injects a `SQLiteEmitter` step that copies its `emit:` schema
and `inputs:` map from the FIRST step whose `address` ends with `"Emitter"`
(capital E). My composite registered the RAM emitter as `"local:ram-emitter"`
(kebab-case). No match → SQLiteEmitter installed with empty `emit:` and
empty `inputs:` → `runs.db` has step-0 rows with `state={}` and nothing else.

The viz auto-renderer then has no observable values to plot. The dashboard
UI showed the framing of each chart but with empty axes — the symptom that
prompted the user's "still" reply with the screenshot.

Fix: change build_document's emitter address to `"local:RAMEmitter"`.

**Recommendation:** Make `inject_sqlite_emitter` match
`addr.lower().endswith("emitter")` OR document the convention explicitly.
A naming convention that's load-bearing but case-sensitive is bad UX.

## 25. viz `inputs()` declaring `'float'` → auto-renderer only passes the LAST scalar

The dashboard's `build_viz_composite` dispatches by port type:

| Declared input type | What gets passed to viz |
|---|---|
| `'list[float]'` | full per-run time series |
| `'float'` | **only `per_run_values[0][-1]` — the last scalar** |
| `'list[list[float]]'` | list of per-run lists |
| (anything else) | `per_run_values[0]` — first run's list |

I had declared `'float'` (the natural per-step scalar type). The viz
received only one data point (the last step) per port. The chart rendered
with axes but no line — the user's "still" screenshot.

Switching to `'list[float]'` makes the auto-renderer pass the full series.
My `update()` must then handle bulk-list input (replace history wholesale)
instead of incrementally appending. I added a `coerce_series()` helper
that handles both forms so the same viz works in both contexts.

**BUT:** declaring `'list[float]'` breaks the *inline* viz Step path —
where the viz is wired into a running composite as a regular Step that
fires every step. The inline path has scalar stores; process-bigraph's
type system refuses to merge `Float` into `List[Float]`:

```
cannot resolve types:
[]: Float(_default=None, _units='', _bits=0)
[]: List(_default=None, _element=Float(_default=None, _units='', _bits=0))
```

So inline viz Step wiring had to be removed. The auto-render path (which
reads runs.db post-run and writes `studies/<name>/viz/*.html`) is now the
only render path.

**Recommendation:** This is a fundamental design tension in the platform.
Either:

1. **Make viz classes polymorphic**: declared inputs report both forms,
   the runtime picks based on caller (incremental vs bulk). Possibly via a
   marker class like `Visualization` already is.
2. **Decouple data-collection from rendering**: viz classes only render
   (always bulk input); the platform handles accumulation. Inline viz
   Steps don't make sense — they'd just buffer in a store the platform
   already buffers in runs.db.
3. **Document the trade-off explicitly**: which port type to declare for
   which use case. Right now the dashboard silently chooses option 4
   (`per_run_values[0]`) when types don't match, which is the worst case.

I went with #2 in practice: removed inline viz Steps from composites
(`_attach_viz_steps` is now a no-op). The user-facing behaviour is the
same — charts appear under `studies/*/viz/` after each run — but the
mental model is simpler.

## 26. `study.yaml.visualizations[]` entries silently 500 without `address:`

The screenshot the user sent showed:
> Failed to render `barrier-kinematics`: KeyError: 'address'

My study yamls listed visualizations as `{name: barrier-kinematics}` only.
The dashboard's renderer requires `{name, address: local:<ClassName>}`.
The error surfaces in the rendered viz HTML (one big iframe per viz, each
with its own error stub) — easy to see in the dashboard, invisible to
`scripts/lint-workspace.py`.

**Recommendation:** Either:

1. Make `address:` required by the workspace schema (currently it's just
   recommended).
2. Auto-resolve `address:` from `workspace.yaml.visualizations[].class`
   when the study only declares `name:`. This is the natural cross-ref:
   workspace.yaml already maps name → class; studies shouldn't have to
   repeat it.

## 27. `runs.db` accumulates rows across reruns; debug runs leave stale rows

Debugging the C++-libc bug, I ran each study several times. Each invocation
appends a fresh row to `runs_meta` and emits whatever the SQLiteEmitter
captures. The auto-renderer reads the *latest* row. If a debug run wrote
`state={}` and a later good run wrote real data, the renderer might
mix them. I worked around with `rm -f studies/*/runs.db` before re-running.

**Recommendation:** `POST /api/study-runs-clear` exists — surface it in the
docs. Or have study-run-baseline take a `clear_previous: true` body field
for fresh-start runs.

## 28. The viz HTML files have unique `div_id` per Visualization-instance

My CouplingTrace produces `<div id="coupling-trace-{id(self)}">`. Two
instances on the same dashboard page (e.g., a study with two baselines)
collide if `id(self)` happens to be the same. Currently safe because each
viz file is its own iframe — but if the dashboard ever inlines them, this
would silently break. Worth fixing now.

**Recommendation:** Use a stable hash (e.g., `hashlib.sha1(class_name +
config['title']).hexdigest()[:8]`) instead of `id(self)`.

## What worked well

- The dashboard's post-run viz-rendering pipeline (`render_visualizations`)
  is well-structured: per-viz iframes, error stubs that don't kill the page,
  observable-name resolution from inputs_map. Once my classes spoke its
  protocol, integration was immediate.
- `scripts/regen-composites.py` saved me from hand-editing 3 YAML files
  every time I changed `build_document` parameters. The "generator function
  + yaml mirror" pattern is the right shape.
- `POST /api/study-run-baseline` is genuinely fast — 3 seconds per study
  for 16 composite steps including all the per-step ReaDDy box-potential
  setup. The pipeline is production-grade once the bugs are out.

## Cross-cutting recommendations

Each log has section-local recs; this log's themes:

1. **Two registries with overlapping names are a recipe for confusion.**
   `composite_discovery` (file-based) and `composite_generator._REGISTRY`
   (decorator-based) should be unified or each renamed to make their
   distinct uses obvious.

2. **C++ stdout pollution is a platform-level concern.** Every model
   package wrapping a native simulator will hit this. Should be a
   first-class feature of pbg-superpowers, not a per-workspace workaround.

3. **Viz inputs() type declarations are load-bearing**, but the
   consequences (bulk vs scalar) aren't documented. Add a section in
   `docs/concepts/visualizations.md` explaining the two paths.

4. **The dashboard makes single-string conventions case-sensitive.**
   `addr.endswith("Emitter")`, `local:ClassName` regex constraints,
   etc. These are quietly load-bearing — surface them or make them
   case-insensitive.
